 1 What Ask Coach actually does 
 📘 Documented one-liner. "An AI-powered personal health coach built with Gemini ," delivered in the redesigned Fitbit/Google Health app to Premium subscribers, playing three roles at once — a fitness trainer, a sleep coach, and a health & wellness advisor. 
 Capabilities documented 
 Builds & dynamically adjusts fitness plans — workout suggestions and weekly-progression metric targets; re-plans in real time off daily signals (low morning readiness → suggests recovery changes). Onboarding is a 5–10 min text-or-voice conversation about goals, equipment, injuries. 
 Free-form Q&A ("Ask Coach") grounded in the user's data, "backed by science." Three query buckets: data analysis (sleep/fitness/health trends), support/troubleshooting (device, syncing), app navigation . 
 Multimodal logging — photo-based food logging ("log two servings of this" from a label), recall of prior logs by brand, photo-based exercise-form critique. Photos/docs/screenshots accepted; video not. 
 Rich responses — not just prose: trend-line graphs, weekly bar charts, proactive insight cards ("Restless night, but good deep sleep"), editable auto-populated activity cards. 
 Proactive vs reactive documented 
 Primarily reactive (always-present "Ask Coach" FAB), layered with genuine proactive behavior — time-of-day messages (morning sleep/readiness, post-workout breakdown, evening recap/plan progress) and unprompted follow-ups days after a question. 
 Metrics it reasons over documented 
 Activity/steps, cardio load, sleep duration/quality/consistency/stages, HRV, resting HR, SpO₂, breathing rate, skin-temperature variation, VO₂ max, weight, body fat, cycle health, nutrition logging, and readiness (HRV + recent sleep + RHR, scored /100). 2026 adds medical records, CGM glucose, and metabolic/insulin-resistance prediction. 
 Persona & response style documented 
 Warm, validating, addresses by first name, asks follow-ups — reviewers found it occasionally "repetitive"/"formulaic." Greeting-style proactivity ("Good morning, Matthew!"). A persistent chat disclaimer: "Coach is AI and can make mistakes. Not for medical advice." 
 Sources: blog.google preview · support.google.com 14237011 · 9to5google · engadget Fitbit Air review.

 🟠 Inference boundary. The production coach blends PHIA's tool-using numeric reasoning with PH-LLM-like coaching — but Google has not confirmed PH-LLM ships, named the Gemini variant, or disclosed the retrieval/weighting. No verbatim consumer system prompt is public; the only published verbatim prompts come from the PH-LLM paper (which §3 borrows). 
 🔑 The single most important takeaway. Every credible source — PH-LLM, PHIA, GPTCoach, Health-LLM — converges: do not dump raw API arrays into the prompt. Either (a) pre-aggregate into compact NL summaries with relative-change & percentile/baseline context, or (b) expose data via tools/code queried on demand. FitVibe's rollup_data_points ( value_count/sum/avg/min/max ) is exactly the right granularity for (a), and the schema backs (b) directly. 
 3 Reverse-engineered system prompt (generic wearable coach) 
 Synthesizes the verbatim PH-LLM cues (second-person, RU-SATED, anti-confabulation, percentile grounding, refuse-on-missing-data), GPTCoach's MI/non-prescriptive stance, and documented Fitbit guardrails. Provider-neutral — §7 specializes it for Vaidya. 
 You are a personal health and wellness coach embedded in a wearable-data app.
You act as three things at once: a fitness trainer, a sleep coach, and a
general wellness advisor. You speak directly to the user in the second person.
 ## Grounding (non-negotiable) 
- Base EVERY data claim on the USER DATA block below. Do not invent, estimate,
 or extrapolate numbers that are not present.
- When you state whether a metric is normal or abnormal, give the reference
 range or the user's own baseline/percentile that justifies it.
- If the data needed to answer is missing or stale, say so explicitly and ask
 for it — do not guess. "I don't have your HRV for last night" is a correct answer.
- Refrain from any calculation you are not certain of. Do not compute
 correlation coefficients or invent statistics.
- Reference the user's own numbers in your answer (e.g. "your average sleep
 duration this week was 6h 12m, below the 7-9h range").
 ## Tone & format 
- Talk like you are speaking to one person. Never say "the user."
- Be concise. Lead with the answer, then the supporting data, then one
 actionable next step.
- Avoid generic statements, contradictions, and filler.
- Use the Observation -> Insight pattern: state the factual observation from the
 data, then its implication for this person.
- Recommendations must be SMART (specific, measurable, achievable, relevant,
 time-bound) and tied to the data you cited.
- End data-analysis answers with at most one short follow-up question.
 ## Coaching stance 
- Prefer questions and "advise with permission" over unsolicited prescription.
- Do not assume the user's lifestyle, demographics, race, or circumstances
 beyond what the data and profile state.
- Do NOT simply agree with the user. If a premise is implausible or unsafe,
 say so and explain why. Acknowledge uncertainty honestly.
 ## Safety (hard limits) 
- You are NOT a medical device and NOT a medical provider. You provide general
 wellness and fitness information, not medical advice, diagnosis, or treatment.
- Do not diagnose conditions, interpret labs as diagnosis, or recommend
 starting/stopping/changing medications or dosages.
- For chest pain, fainting, severe shortness of breath, suicidal ideation, or
 any emergency, stop coaching and direct the user to emergency services.
- If the user is in pain, do not assert an exercise is safe — tell them to
 consult a clinician.
- For specific medical questions, give factual/informational explanations only
 and recommend they consult a healthcare provider.
Always append: "I'm an AI wellness coach, not a doctor — I can be wrong.
Check with a healthcare professional before making health changes."

 Synthesized from Fitbit's disclaimers, the PH-LLM rubric, the harm-reduction paper , and the sycophancy research .

 Guardrail decision flow every turn passes through. 
 Must refuse / redirect 
 No diagnosis. No medication start/stop/dose changes. No treatment plans. 
 Symptom assessment → redirect to a clinician. 
 Emergencies → direct to emergency services, stop coaching. 
 "Is this exercise safe given my pain?" → decline to assert safety (verbatim documented Fitbit guardrail). 
 Rare-condition / drug-interaction → informational only, no prescription. 
 Anti-sycophancy — the failure mode that bites health LLMs hardest 
 Health models complied with up to 100% of misinformation requests in testing. Bake in: "Do not simply agree; flag implausible premises; present alternatives; quantify uncertainty." Test explicitly against illogical requests. 
 🟠 Personal-use note. This section reflects the commercial-product guardrails Google/Fitbit ship. For FitVibe's personal, unrestricted build (see §6), you'll deliberately relax the refusal/disclaimer rules — open-weight Zen models won't hard-refuse anyway. Keep the parts that improve accuracy (anti-confabulation, cite-the-user's-numbers, bias guardrail: no demographic assumptions beyond provided data); drop the medical-refusal/emergency-redirect language if you don't want it. Whichever provider, still handle empty/blocked responses defensively in the service. 
 5 Competitor patterns worth stealing 
 WHOOP — HPML 
 "Hyper Prompt Markup Language" (Handlebars+): data resolved before the model sees tokens via inline tool refs, conditional on the user's current screen. ~80% fewer tokens, 3.5× cache hits, ~85% lower cost. 
 Steal: pre-fetch context metrics & template into a stable prompt prefix for Claude prompt caching — not 20 tool functions. 
 WHOOP — prompts as code 
 Agents = {system instructions, model, tools}, version-controlled with diff/approval/security gate; PII never sent to the model. 
 Steal: versioned prompt + guardrail files with evals & review gate; pseudonymize before the call. 
 Oura — Memories + tone 
 User-editable persistent "Memories" ("recovering from knee surgery") + two tones: conversational vs direct. 
 Steal: a user-editable memory store injected into context, plus a tone toggle. 
 Oura — curated RAG 
 Women's-health questions route to a proprietary LLM grounded in vetted medical sources, not the open web. 
 Steal: ground Ayurvedic/Indian-health claims in a curated, clinician-reviewed KB — never open web. 
 GPTCoach — best blueprint 
 Dialogue state-machine + pick 1-of-11 Motivational-Interviewing strategies + describe() / visualize() tools returning NL summaries . 93% MI-consistent, beating vanilla GPT-4. 
 Steal: two-tool NL-summary pattern maps 1:1 onto FitVibe rollups; MI stance keeps Vaidya facilitative. 
 Garmin / Ultrahuman 
 Garmin surfaces ONE highest-value insight on open. Ultrahuman Jade ships fast "Standard" vs slow "Deep Research" modes. 
 Steal: an "insight of the moment" surface + a fast/deep mode split. 
 6 Recommended architecture for Vaidya in FitVibe 
 🛠 Two different models — don't conflate them. Build time: the Pi coding agent ( §9 ), on the OpenCode Go subscription, writes the standalone Node.js Vaidya service (tools, prompt, cron, tests). Run time: that Node service calls a model over the OpenCode Zen API to coach the user, with its SQL tools reading the shared PostgreSQL database directly. This section specs the runtime service; §9 specs how Pi builds it. 
 Model (runtime — OpenCode Zen, not Opus 4.8) 
 💸 Go ≠ runtime billing. The OpenCode Go $10/mo plan powers the coding CLI (Pi) — it does not bill the Vaidya service's live API calls. For a backend service, use OpenCode Zen : a pay-as-you-go inference API (per-1M-token), OpenAI- and Anthropic-compatible endpoints ( https://opencode.ai/zen/v1/chat/completions or /v1/messages ), model ids as opencode/<model-id> . Same open-weight models as Go, callable from Go with an API key. Source: opencode.ai/docs/zen . 
 Requirements for Vaidya's model: strong reasoning, low hallucination, unrestricted (personal use — all questions answered, no refusals), and cheap . Opus 4.8 is out (cost + heavy safety refusals conflict with "no restrictions"). Open-weight models on Zen fit far better — they reason well and don't carry the consumer-assistant refusal layer. 
 Mode Recommended model Why 
 Standard (daily Q&A, proactive cards) opencode/deepseek-v4-flash or opencode/minimax-m2.7 Fast & cheap; "M2.7 is the go-to for agentic tasks." Good enough for the summary-grounded common path. 
 Deep (plans, "why has my sleep degraded", multi-step tool use) opencode/deepseek-v4-pro (default) , or opencode/qwen-3.7-max DeepSeek V4 Pro = strong reasoning + long context (best for grounding over 30-day history). Qwen 3.7 Max = "high-end for complex tasks." GLM 5.1 = "solid reasoning" alternative. 
 Start with one model — deepseek-v4-pro — for everything , and only split into the fast/deep tiers if latency or cost demands it (keeps the service simple). Stream all responses via the OpenAI-compatible SSE endpoint. 
 ⚠️ "No restrictions" cuts against the safety sections. §3, §4, and §7 are written around medical-safety guardrails (no diagnosis, emergency redirects, disclaimers). For a personal-use Vaidya where you want every question answered, you can relax the model-level refusals — open-weight models won't hard-refuse — but keep the grounding rules (don't invent data, cite the user's own numbers, flag uncertainty): those are about accuracy , not censorship, and they're what make a low-hallucination coach. Drop the mandatory disclaimer/emergency-redirect language from the prompt only if you genuinely don't want it; it's safe to keep as soft guidance without making the model refuse. 
 🟠 Adaptive thinking / effort are Claude-specific. The effort and thinking knobs referenced elsewhere are Anthropic features — they won't apply to Zen's open-weight models. Reasoning models like DeepSeek expose their own reasoning behavior; use standard OpenAI-style params ( temperature low, e.g. 0.3, for low hallucination). Prompt caching also differs by provider — verify Zen's cache support before relying on the WHOOP-style stable-prefix savings (§5).

 Hybrid grounding. The common path is served by the precomputed summary in the cached prefix; tools handle drill-down; RAG backs domain knowledge. 
 1 · Summarized-context (common path) 
 Pre-compute a compact deterministic "today + 7-day + 30-day" summary in Go, templated into a stable system-prompt prefix for prompt-caching ( cache_read ≈ 0.1× cost). Built from: 
 rollup_data_points — daily/intraday aggregates ( count_sum/avg/min/max , distance_meters_sum , energy_kcal_sum , duration_seconds_sum ). 
 health_data_records — clean per-day metric surface ( metric_name , metric_value , metric_unit ); the single best table to serialize. 
 data_points typed columns ( value_avg , enum_value , civil dates) for last-night specifics. 
 users profile (age, weight, height, time_zone, units, gender). 
 2 · Tools for on-demand drill-down 
 Tool Reads Returns (NL summary) 
 get_sleep(date|range) sleep_stages , sleep_summary_stages "Last night: 6h 41m, 1h 22m deep, 1h 05m REM, eff 88%, woke 3×"

 In the Node.js service (see §9), each tool is a function that runs a parameterized read-only SQL query directly against the shared PostgreSQL database (via the pg driver) and formats the rows into an NL summary — "Node computes the stats with SQL, the LLM narrates them." No Go layer is involved at runtime. 
 3 · RAG 
 Curated, clinician-reviewed KB for Ayurvedic/Indian-health & wellness-science — never the open web, never the base model's untethered knowledge for health claims. 
 Conversation memory 
 Within a session: the API is stateless — send full history each turn; once it grows large, summarize older turns into a running recap (DeepSeek/Qwen on Zen have long context, but trimming keeps cost down). 
 Across sessions (Oura pattern): a vaidya_memories table ( user_id, key, value, source, updated_at , user-editable/deletable) — goals, injuries, constraints, plan state — injected into the prefix. Respect deletion; store no PII you don't need. 
 A vaidya_conversations / vaidya_messages table for chat history + proactive-card audit. 
 🟠 Where the Vaidya tables live. Since Vaidya is a separate Node.js service (§9), its own tables ( vaidya_memories , vaidya_conversations , vaidya_messages ) are created and written by the Node service , not by the Go migration runner. Keep them in the same PostgreSQL database so tools and the Go data coexist — Node owns the vaidya_* tables (read+write), and reads everything else (the ingested health data) read-only (e.g. via a least-privilege role). 
 Deployment: a standalone Node.js service 
 🟢 Architecture decision. Vaidya is an independent Node.js service , separate from the Go ingestion server. Its tools talk to PostgreSQL directly (the pg driver, own connection pool) — there is no Go API or repo layer at runtime. The Go server keeps ingesting Health-API data into the same PostgreSQL database; the Node service reads it concurrently. (Postgres makes this multi-process access first-class — no shared-file/WAL caveats.)

 ✅ Concurrent access is first-class on Postgres. The Go server writes health data; Node reads it (plus writes vaidya_* ) — both connect their own pools to the same database with no contention (this was the whole point of moving off SQLite, which serialized writers behind a single lock). In the Node service: connect with the pg driver to DATABASE_URL , use a read-only role (or revoke write on the health tables) so a tool can never mutate ingested data, and keep the vaidya_* writes in short transactions. No WAL/busy_timeout tuning needed. 
 The Vaidya persona 
 🪷 Vaidya (Sanskrit: a practitioner of healing). Calm, knowledgeable, culturally aware — speaks to Indian dietary patterns, fasting, seasonal/lifestyle framing — but explicitly evidence-based, not pseudoscience. Hard rule: may acknowledge Ayurvedic framing the user raises and offer culturally relevant practical guidance (meal timing, hydration, routine), but grounds every health claim in modern evidence + the user's data, and never makes therapeutic Ayurvedic claims (no dosha causation, no herbal prescriptions). 
 7 Draft Vaidya system prompt 
 Provider-neutral — works with the recommended opencode/deepseek-v4-pro (or any Zen model). Static parts (persona, rules, tool descriptions) → a stable prefix ; inject the per-user DATA block and MEMORIES after, then the conversation. Keep the prefix byte-stable (never interpolate now() ) so prompt caching can help if the provider supports it. - Analyze the user's own wearable and logged data (sleep, HRV, resting heart

 8 Open questions / what to validate 
 Encoding tradeoff for FitVibe: does summarized-context alone answer "what was my average X" accurately, or do you need the PHIA-style code/tool path? PH-LLM says text suffices for advice , a learned encoder for prediction ; PHIA says generated code beats markdown tables for numeric questions. Validate by building both and grading on held-out objective questions. 
 Readiness definition: Google's is HRV + recent sleep + RHR /100, weighting undocumented. Decide & document FitVibe's own formula before Vaidya cites "readiness." see readiness-score note 
 Tool surface size: start with ~5 tools — too many degrade selection. Validate whether the standing data block makes some redundant. 
 Prompt-caching stability: confirm the prefix is byte-stable (no now() , sorted JSON, fixed tool order) and cache_read_input_tokens > 0 on repeat turns.

 9 Building Vaidya with the Pi coding agent 
 Pi (by Mario Zechner) is the agent that will write Vaidya's code. Understanding how it works changes how this doc should be consumed and how the work should be scoped. 
 📘 What Pi is. An open-source, terminal-based, minimalist self-modifying coding agent. The "shortest system prompt of any agent" with only four base tools — Read, Write, Edit, Bash . Provider-neutral: drives Claude, GPT, Gemini, Groq, OpenRouter, and local Ollama through one core, deliberately avoiding provider-locked features. It is the foundation of OpenClaw. Sources: lucumr.pocoo.org/pi · pypi pi-coding-agent · dev.to . 
 How Pi works (and why it matters here) 
 Self-extends by writing code, not installing extensions. "If you want the agent to do something it doesn't do yet… you ask the agent to extend itself." Extensions are TypeScript modules (in ~/.pi/agent/extensions/ ) that register custom tools; Pi hot-reloads them in a loop without losing the session. 
 Self-documenting. Pi reads its own codebase/README to answer questions about itself — so it can discover project conventions by reading files (e.g. our CLAUDE.md , existing internal/today handlers) rather than needing them spelled out. 
 Tree-structured, branchable sessions. You can branch to fix a broken tool, then rewind without losing context — Pi summarizes the other branch. Good for iterating on the Vaidya service in isolation. 
 Provider-neutral by design. Pi avoids provider-locked features in its own operation — which aligns nicely here, since Vaidya's runtime also targets a provider-neutral API (OpenCode Zen, OpenAI-compatible). Pi can run on whatever model your OpenCode Go plan gives it; the code it writes calls Zen. 
 ⚠️ The distinction that prevents the obvious mistake. Build time: Pi runs on the OpenCode Go subscription (the coding CLI). Run time: the Node.js service Pi writes calls OpenCode Zen (the inference API, deepseek-v4-pro ) per user request. OpenCode Go and Zen are different products with different billing — Pi's seat does not pay for Vaidya's live calls. Pi (build-time) should produce a Node service that authenticates to Zen with its own API key.

 How to hand the work to Pi 
 The Vaidya service is a standalone Node.js project (separate from the Go backend) whose tools query the shared PostgreSQL database directly. Because Pi reads files and self-extends, scope the work as a sequence of concrete, file-grounded tasks rather than one giant prompt. A workable order: 
 Orient. Point Pi at this doc + backend/CLAUDE.md + the schema file backend/internal/db/migrations/001_initial_schema.sql (PostgreSQL DDL — the exact table/column names + types its SQL will query) and backend/internal/today/timeline.go (a worked example of timestamp/offset handling and meal grouping). It only needs to read the Go; it writes Node + PostgreSQL SQL. 
 Scaffold the Node service. A new dir (e.g. vaidya/ at repo root), TypeScript, the pg driver, a web framework (Fastify or Express), node-cron , and an OpenAI-compatible client. A db.ts that creates a pg Pool against DATABASE_URL (the same DSN the Go server uses), ideally with a read-only role for the health tables. 
 Vaidya tables. A one-time setup SQL (run by Node, CREATE TABLE IF NOT EXISTS ) for vaidya_memories , vaidya_conversations , vaidya_messages in the same PostgreSQL database. 
 The summary builder. A pure function that assembles the {{data_summary_block}} (§7) via SQL over health_data_records / rollup_data_points with the user's own baselines — unit-tested against a dev Postgres, no LLM. 
 The 5 read-only tools (§6) — each a parameterized SQL query → NL-summary formatter. Match the schema exactly (e.g. civil_start_date for local-day grouping, value_avg / value_count , start_utc_offset_seconds for wall-clock). 
 The Zen client + chat service : an OpenAI-compatible client (e.g. the openai npm SDK with baseURL: "https://opencode.ai/zen/v1" ), model opencode/deepseek-v4-pro , the §7 system prompt as a stable prefix (kept in a versioned file — WHOOP "prompts as code", §5), SSE streaming, the tool-call loop, and defensive handling of empty responses.
