FitVibe · Database Migration Plan

SQLite (Turso/libSQL) → PostgreSQL

A full migration of the Go ingestion backend off SQLite onto PostgreSQL, driven by the move to multi-process concurrency (Go ingester + Node "Vaidya" coach + the app all hitting one DB). Includes promoting hot JSON fields to real columns at ingestion time so queries become plain column reads.

Driver: pgx (jackc/pgx v5) Target: PostgreSQL 16/17 (latest stable) Data: fresh start + re-backfill Host: office server

0Why this migration (and the honest caveat)

🎯
The real driver: concurrency. The architecture is becoming multi-process — the Go ingester (writer), the Node Vaidya service (reader + writer of vaidya_*), and the app. SQLite serializes all writers behind a single write lock; under that contention you get database is busy errors (we saw them when a backfill writer overlapped webhook writers). PostgreSQL is built for concurrent multi-process access with row-level MVCC — it removes that whole class of problem. Hosting on the office server (not the Pi) removes the only real downside of self-hosting.
🟠
Caveat worth stating once. Most of the recent data bugs (missing nutrition, inflated steps, stuck refresh) were not SQLite being unreliable — they were upstream API lag, query logic, and a missing HTTP timeout. Postgres would not have fixed those. So this migration is justified by concurrency + future architecture, not by SQLite being buggy. Going in with the right expectation matters: it buys clean multi-writer access, not magically correct data.

1Driver & connection layer medium

Replace the tursogo driver + the pragmaConnector (which exists only to run SQLite session PRAGMAs on every pooled connection) with pgx v5 (github.com/jackc/pgx/v5, via pgxpool or the stdlib database/sql adapter).

SQLite today (internal/db/db.go)PostgreSQL
tursogo.NewConnector(url)pgxpool.New(ctx, dsn) or sql.Open("pgx", dsn)
pragmaConnector running PRAGMAs per-connDelete entirely — see below
PRAGMA foreign_keys = ONFKs always enforced — remove
PRAGMA busy_timeout = 5000Pool + statement_timeout / lock_timeout — remove PRAGMA
PRAGMA journal_mode = WALN/A (Postgres has its own WAL) — remove
PRAGMA encryption_keyTLS in transit + disk encryption / pgcrypto if needed — remove from code
✅
This is mostly deletion. The whole pragmaConnector dance disappears; db.Open becomes "open a pool against a DSN, ping, run migrations." Connection pooling that we hand-tuned for SQLite's single-writer reality is now the database's job.

The rest of the codebase uses the standard database/sql interface (QueryRowContext, QueryContext, ExecContext, sql.NullX), all of which pgx supports unchanged — so the change is concentrated in db.go plus the SQL-dialect fixes in §5.

🕐
Timestamp gotcha that bit us before. We hit a bug where sample_time stored local wall-clock mislabeled as UTC. With TIMESTAMPTZ, be explicit: store true UTC instants, keep the separate *_utc_offset_seconds columns for local rendering (as we already do), and make the mapper write the real physicalTime (the fix already landed). Postgres will normalize TIMESTAMPTZ to UTC on store, which is what we want.

payload_json as JSONB (not TEXT) is a free upgrade: the raw payload stays lossless, but if we ever need an ad-hoc field we haven't promoted to a column, it's queryable/indexable — no re-ingest needed.

Generic vs specific. 4The unique-index dedup — the one genuinely tricky bit hardest

The upsert dedup contract is the heart of the system (overlapping fetch windows are safe because of it). SQLite implements it with COALESCE(col, '') in a unique index, exploiting SQLite's loose typing (a timestamp coerced to '' when NULL). Postgres won't coerce a timestamptz to '', and treats NULLs as distinct in unique indexes by default — so the naive port breaks dedup.

SQLite today

-- idx_data_points_unique (migration 002)
CREATE UNIQUE INDEX idx_data_points_unique ON data_points (
  user_id, data_type,
  COALESCE(sample_time, ''), COALESCE(start_time, ''), COALESCE(end_time, '')
);

PostgreSQL (latest stable) — clean fix with NULLS NOT DISTINCT

-- PG 15+ treats NULLs as equal in the unique index, so two rows with the
-- same (user, type) and both NULL start_time collide as intended.
CREATE UNIQUE INDEX idx_data_points_unique ON data_points (
  user_id, data_type, sample_time, start_time, end_time
) NULLS NOT DISTINCT;

And the upsert's conflict target uses the same column list (no COALESCE):

INSERT INTO data_points (...) VALUES (...)
ON CONFLICT (user_id, data_type, sample_time, start_time, end_time)
DO UPDATE SET ...;
⚠️
Three checks for the implementer. (1) ON CONFLICT with NULLS NOT DISTINCT indexes is supported on PG 15+ — confirm the target list exactly matches the index columns. (2) Same treatment for idx_rollup_data_points (migration 003) — drop its COALESCE form, rebuild with the real columns + NULLS NOT DISTINCT. (3) Migration 003 also has a one-time DELETE … GROUP BY COALESCE(...) dedup cleanup — not needed on a fresh DB; drop it.

The InsertMany path preserves rowids today via ON CONFLICT DO UPDATE (not INSERT OR REPLACE) so child FKs stay valid. Postgres keeps the row identity the same way — but the two INSERT OR REPLACE sites (rollup_data_points.go:88, health_data_records at data_points.go:389) must be rewritten to ON CONFLICT … DO UPDATE to preserve identity (Postgres has no INSERT OR REPLACE).

✅
Two bugs disappear for free. The parseStoredTime helper existed only because SQLite's COALESCE dropped TIMESTAMP affinity and returned strings — Postgres keeps the type, so we scan straight into time.Time and delete the helper. And the json_extract sites vanish entirely once fields are promoted (§3).
🟠
Placeholder rewrite is mechanical but error-prone. Numbering $1…$N by hand across 40 statements invites off-by-one bugs. Mitigations: (a) use squirrel or pgx named args, or (b) keep raw SQL but add a test per repo that executes each query against a throwaway PG instance. Recommend the latter — minimal dependency, catches every placeholder/typing error.

6Config & commands easy

  • config.go: replace TursoDatabaseURL / TursoEncryptionKey / SQLiteBusyTimeoutMs with DatabaseURL (a standard postgres://user:pass@host:5432/fitvibe?sslmode=… DSN) and pool knobs (DB_MAX_CONNS, etc.). Keep the env-tag pattern.
  • .env.example / .env: swap the SQLite lines for the DSN.
  • .gitignore: the fitvibe.db* ignores become irrelevant (harmless to leave).
  • cmd/server, cmd/backfill, cmd/fetchbackfill, cmd/webhooks: all call db.Open(cfg, logger) — no change beyond the new signature/DSN.
  • The locking constraint disappears: today fetchbackfill requires the server stopped (single-writer lock). With Postgres, fetchbackfill and the server can run concurrently — a real quality-of-life win we hit repeatedly this week.

7Vaidya (Node.js) service impact medium

🔑
This migration improves the Vaidya design. The Vaidya doc flagged a real risk: two processes on one SQLite file, safe "only because of WAL," with a warning to avoid long write transactions that block ingestion. Postgres erases that risk — Vaidya just opens its own pool to the same Postgres database; concurrent reads + its own vaidya_* writes are first-class, no shared-file coupling.

Changes to the (not-yet-built) Node service spec in vaidya-coach-research.html:

  • better-sqlite3 → pg (node-postgres) or postgres.js. Connection is a pooled DSN, not a file path.
  • Tool SQL: ? → $1, json_extract → promoted columns (same as Go), date() → ::date.
  • vaidya_memories / vaidya_conversations / vaidya_messages live in the same Postgres DB; Node creates them (or a shared migration tool does).
  • The "WAL / busy_timeout / read-only / no long transactions" caveat section becomes obsolete — replace with standard PG pool config.

8Rollout steps

  1. Stand up Postgres on the office server (latest stable). Create the fitvibe DB + a least-privilege app role. Confirm reachable from where the Go server runs (network/firewall/sslmode).
  2. Driver swap in db.go: pgx pool, delete pragmaConnector, new db.Open. Keep the embedded-migrations runner (it just Execs .sql in order).
  3. Rewrite the 3 migration files as Postgres DDL: identity PKs, TIMESTAMPTZ, JSONB, the new promoted columns (§3), and the NULLS NOT DISTINCT unique indexes (§4). Drop the one-time dedup DELETE.
  4. Mapper: add extractScalars/children cases to populate the new columns (carbs, fat, meal_type, food_display_name).
  5. Repo SQL rewrites (§5): placeholders, RETURNING, ON CONFLICT, drop json_extract/parseStoredTime. One query-execution test per repo against a disposable PG.
  6. Config: DSN env, pool knobs, .env.example.
  7. Build + test: go build ./..., go test ./...; bring up the server against empty Postgres; verify migrations apply idempotently.
  8. Re-backfill: run fetchbackfill to repopulate from the Health API — now fills the promoted columns from the start.
  9. Smoke-test the endpoints (today summary, nutrition, timeline, sleep) against real repopulated data; confirm step de-dup, nutrition columns, timeline grouping all match.
  10. Update the Vaidya doc (§7) so the future Node service targets Postgres, not SQLite.
