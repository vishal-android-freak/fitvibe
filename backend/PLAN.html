Fitvibe Backend — Implementation Plan

Google Health API ingestion backend running on a Raspberry Pi, backed by local Turso.

1. Goal

Build a Go backend that runs on a Raspberry Pi and persists Google Health API data into a local Turso (libSQL) SQLite database. Data ingestion happens via:

  1. Webhooks for Google Health data types that support real-time notifications.
  2. Cron jobs for data types/sources that do not support webhooks, plus aggregation/rollup data.

An Expo mobile app will initiate a Google OAuth flow and hand the offline authorization code to the backend, which exchanges it for access/refresh tokens and stores them. The backend will expose REST APIs that the app uses to fetch stored health data. The backend is also responsible for ingestion, persistence, and later AI-agent writes via an MCP server.

3.2 Webhook-supported vs Cron-only Data Types

The DataPoint.data union contains 37 data types. calories-in-heart-rate-zone and total-calories are rollup-only and do not appear as raw payloads.

Webhook payloads contain only metadata — healthUserId, dataType, operation, and time intervals. The backend must still call :list to fetch the actual data values. Historical electrocardiogram data may lack timezone offsets.

6. Webhook Implementation

6.1 Endpoint

POST /webhooks/google-health — configurable via env vars.

6.2 Verification Handshake

  1. Google sends {"type": "verification"} with Authorization → respond 200 OK.
  2. Google sends the same body without credentials → respond 401 Unauthorized.

6.3 Signature Verification

Verify GOOGLE-HEALTH-API-SIGNATURE header against raw JSON body using Tink PublicKeyVerify. Public keyset URL:

https://www.gstatic.com/googlehealthapi/webhooks/webhooks_public_keyset.json

7.3 Sync Window

On first OAuth exchange, a background backfill job automatically fetches the last DEFAULT_BACKFILL_DAYS (default 30) for all data types.

8. OAuth / Token Flow

8.1 Expo App

  1. Initiate Google OAuth with all health/profile scopes, access_type=offline, and prompt=consent.
  2. Redirect URI is the app scheme, e.g. com.fitvibe.app:/oauth2callback.
  3. Send the authorization code to backend POST /auth/exchange.
  4. Fetch health data through backend REST APIs; the app does not call Google Health directly.

8.2 Backend /auth/exchange

POST /auth/exchange
{
  "code": "...",
  "redirect_uri": "com.fitvibe.app:/oauth2callback"
}
  1. Exchange code with Google token endpoint.
  2. Call GET /v4/users/me/identity to get healthUserId.
  3. Encrypt access + refresh tokens with AES-256-GCM.
  4. Upsert users row.
  5. Trigger automatic backfill job.

11. Implementation Phases

Phase 1 — Foundation

  • Set up Go module, structured packages, and configuration loader.
  • Add Turso/libSQL Go driver (tursogo) in local-only mode and migration runner.
  • Configure local Turso/libSQL database encryption at rest (tokens protected by DB encryption, no separate app-level encryption).
  • Implement Google OAuth code exchange and token refresh.

Phase 2 — Database & Models

  • Write migration files for all tables.
  • Rely on libSQL database encryption for the local Turso database file.
  • Enable libSQL/SQLite features: foreign keys, WAL journal mode. Use triggers and views where helpful.
  • Set PRAGMA journal_mode = 'wal' and PRAGMA busy_timeout for concurrent webhook + cron writes.
  • Implement DB models and CRUD helpers. Add users, sync_state, webhook_notifications repositories.

Phase 3 — Google Health API Client

  • Build typed HTTP client for health.googleapis.com/v4.
  • Implement list, reconcile, get, rollUp, dailyRollUp.
  • Implement getIdentity, getProfile, getSettings.
  • Implement pagination and rate-limit backoff.

Phase 4 — Webhooks

  • Implement verification handshake handler.
  • Implement signature verification with Tink.
  • Implement async notification processor with SQLite-backed queue.
  • Implement subscriber management admin endpoints.

Phase 5 — Cron Sync

  • Add cron scheduler.
  • Implement generic list-sync job for cron-only data types.
  • Implement rollup/daily-rollup sync jobs.
  • Implement profile/settings sync job.
  • Implement backfill job triggered on new user signup.
  • Configure incremental backfill and cron sync for local Turso.

Phase 6 — API Surface

  • Wire HTTP router.
  • Add /auth/exchange, /auth/refresh, /users/me.
  • Add query endpoints for the Expo app (/v1/data/:dataType, /v1/rollup/:dataType, /v1/summary/:metric).
  • Add admin endpoints.
  • Add /healthz and structured logging.

Phase 7 — Hardening

  • Request logging, panic recovery, timeouts.
  • Idempotent webhook processing and dead-letter handling.
  • Metrics/health endpoints.
  • Tests for critical paths.

Phase 8 — Raspberry Pi Deployment

  • Build Linux ARM64 binary.
  • Add systemd service file.
  • Expose webhook endpoint via Cloudflare tunnel or ngrok.
  • Document tunneling setup.
